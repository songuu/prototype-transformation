# 阶段5：代码审计

## 目标

以阶段4完整需求文档为标准，审计代码实现是否完整正确地覆盖了所有需求，同时发现代码层面的 Bug。

> 主 CLI 直接调用 Codex，不需要 Opus。

---

## 两个审计维度

```
维度1：需求偏差检测              维度2：Bug 检测
──────────────────            ──────────────
□ 功能点是否全部实现？           □ 异步 loading/error 处理
□ 交互逻辑是否与文档一致？        □ 表单校验与文档规则一致？
□ 字段展示是否与文档一致？        □ 空状态是否处理？
□ 状态流转是否正确？             □ 文件上传大小/格式校验？
□ 接口调用路径/参数/字段一致？    □ 分页边界情况？
□ 错误处理是否按文档实现？        □ 权限校验是否遗漏？
```

---

## 输入

| 项目 | 说明 |
|------|------|
| 需求文档 | 阶段4人工确认后的版本 |
| 代码仓库路径 | 本地路径，如 `src/pages/knowledge-base/` |
| 审计范围（可选） | 不指定则审计需求文档覆盖的所有相关文件 |

---

## git diff 决策

| 场景 | 传什么 |
|------|--------|
| 首次审计 | **完整文件**（每批 ≤ 5 个文件） |
| 修复后复审 | **git diff** |
| 修复 > 50 行或跨多文件 | **完整文件 + diff 摘要** |

---

## 执行步骤

### 1. 确定范围，按模块分批

```
需求文档页面 → 代码文件
知识库列表页 → src/pages/knowledge-base/index.tsx
创建弹窗     → src/components/CreateKBModal.tsx
```

每批 ≤ 5 个文件，按页面或功能模块分组。

---

### 2. Codex 审计 Payload

调用配置见 `references/Codex调用协议.md`（全新会话，high；多批次 resume 同一会话）

```
你是一位资深前端工程师，对代码实现做双维度审计。

【需求文档标准（The Standard）】
[阶段4完整需求文档]

【当前批次】第 N 批，共 M 批
【审计文件】
--- [文件路径] ---
[完整代码]

【维度1：需求偏差检测】
- 需求文档中的功能点是否在代码中实现？
- 交互逻辑是否与文档一致？
- 列表/表单字段是否与文档字段定义一致？
- 接口调用是否与文档绑定的接口一致（路径、参数、字段名）？
- 错误处理是否按文档描述的异常场景实现？

【维度2：Bug 检测】
- 异步操作是否有完整的 loading/error 处理？
- 表单校验是否与文档字段规则一致？
- 列表为空时的空状态是否处理？
- 文件上传的大小/格式校验是否实现？
- 分页边界情况是否正确？

返回格式：
[文件/行号] - [需求偏差/Bug] - [CRITICAL/HIGH/MEDIUM/LOW] - [问题] - [建议]

当前批次无问题：当前批次 CLEAN
```

### 5. 修复后复审

用户回复「复审」后，resume Codex 会话传 git diff：

```
已完成修复：
- [文件:行]：[修复摘要]

git diff：[内容]

请确认修复是否正确，检查是否引入新问题。
```

---

## 边界处理

| 场景 | 处理方式 |
| ------ | --------- |
| 代码路径不存在 | 提示用户确认路径，列出可能的候选目录 |
| 代码文件超过 20 个 | 按模块分组，预估批次数，询问用户是否全部审计 |
| 需求文档不完整（跳过了阶段3/4） | 提示用户补充，或基于现有内容审计并在报告中标注局限 |
| Codex 不可用 | 停止，告知用户，禁止自我替代 |
| 复审时 git diff 为空 | 提示用户确认是否已提交修复，或是否需要重新审计 |

---

## 反模式

- ❌ 首次审计传 git diff（缺少需求文档对比上下文）
- ❌ 一次传超过 5 个文件
- ❌ 只做 Bug 检测，不做需求偏差检测
- ❌ 修复后不复审，自行宣布通过
- ❌ 报告产出后自动修复，不等人工确认
