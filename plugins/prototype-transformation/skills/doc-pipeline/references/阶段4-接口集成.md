# 阶段4：接口集成

## 目标

将中间文档每个功能点与后端接口绑定，有接口直接用，没接口提供 Mock 代码，让前端开发拿到文档就能开始工作。

---

## 输入

| 项目 | 格式 |
|------|------|
| 中间文档 | 阶段3人工确认后的版本 |
| 接口文档 | Swagger/OpenAPI JSON或YAML / 手写说明 / 直接粘贴 |

三种格式可混合使用。

---

## 执行步骤

### 1. 主 CLI 委派 Opus 子代理

```
任务：接口文档解析与需求绑定
中间文档：[完整内容]
接口文档：[Swagger内容 / 手写文档 / 粘贴内容]
要求：
  1. 解析所有接口，整理为接口清单
  2. 将每个功能点与对应接口绑定（三类缺口处理）
  3. 前台调用 Codex xhigh 审查一致性（详见 Codex调用协议）
  4. 处理反馈后产出最终文档，交回主 CLI
```

### 3. Opus — 三类缺口处理

**类型 A：有接口且完整** → 直接绑定

```markdown
### 创建知识库

> 🎨 设计稿：[Figma →](链接)  
> 🔌 **接口：** `POST /api/knowledge-base`

**请求参数**
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 最大100字 |

**响应**
\```json
{ "id": "string", "name": "string", "createdAt": "ISO8601" }
\```

**错误码**
| 码 | 说明 |
| 400 | 参数校验失败 |
| 409 | 同名已存在 |
```

**类型 B：有接口但不完整** → 标注缺失

```markdown
> 🔌 **接口：** `GET /api/dataset/:id/status`  
> ⚠️ **待完善：** 响应中 `status` 字段缺少枚举值定义，已记录至附录B。
```

**类型 C：无接口** → 提供 Mock 方案

```markdown
> 🔴 **暂无接口** — 建议方案：

**方案1：前端 Mock（推荐）**
\```typescript
// mock/folder.ts
export const mockFolderCreate = async (data: FolderCreateParams) => {
  await delay(300)
  return { id: generateId(), name: data.name, createdAt: new Date().toISOString() }
}
\```

**方案2：等待后端接口**
预计路径：`POST /api/folder`  字段：`name`（必填）、`description`（可选）
```

### 4. Codex 一致性审查 Payload

调用配置见 `references/Codex调用协议.md`（全新会话，xhigh）

```
你是一位全栈架构师，审查需求文档与接口文档的对应关系。
目标是发现接口定义与需求之间的不一致。

【需求文档（含接口绑定草稿）】[完整内容]

请审查：
1. 接口覆盖度 — 有功能点缺少接口但文档未标注的？
2. 字段一致性 — 接口返回字段是否能支撑所有展示需求？
   （如列表需显示"创建人姓名"但接口只返回"创建人ID"）
3. 接口设计风险 — 哪些接口会导致前端多次请求才能完成一个操作？
4. 错误处理覆盖 — 需求中的异常场景是否都有对应错误码？
5. Mock 方案评估 — Mock 方案是否能覆盖所有测试场景？

返回格式：
[功能点/接口] - [CRITICAL/HIGH/MEDIUM/LOW] - [问题描述] - [建议]
```

## 边界处理

| 场景 | 处理方式 |
| ------ | --------- |
| 接口文档为空或极少 | 所有功能点标注为类型C，全部提供 Mock 方案 |
| Swagger JSON 解析失败 | 提示用户检查格式，尝试手动解析或请用户粘贴文本 |
| 接口版本冲突（v1/v2 并存） | 询问用户使用哪个版本，或两者都标注 |
| Codex 不可用 | 跳过一致性审查，在确认提示中说明，由人工 review 替代 |

---

## 反模式

- ❌ 字段不一致时不标注，直接绑定
- ❌ 无接口的功能点不提供 Mock 方案
- ❌ 主 CLI 自行解析，不委派 Opus
- ❌ 产出后自动进入阶段5
