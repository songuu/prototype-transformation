# 阶段2：原型解析

## 目标

将原始需求文字 + 原型图，转化为结构化初始文档。  
核心不是"照着图描述"，而是**主动挖掘图里没画出来但必须有的内容**。

---

## 输入

| 项目 | 格式 | 说明 |
|------|------|------|
| 原始需求文档 | 文字 / PDF / MD | — |
| 原型图 | 多张 PNG/JPG | 建议按页面命名：`01-列表页.png`、`02-创建弹窗.png` |

图片未命名时按上传顺序编号，推断页面名称。

---

## 执行步骤

### 1. 主 CLI 委派 Opus 子代理

```
任务：原型图分析与初始文档生成
原始需求文档：[完整内容，原文传入]
原型图：[已上传图片，按顺序标注编号]
要求：
  1. 逐页分析，提取界面元素和交互逻辑
  2. 按补全检查清单，主动补全图中未体现的内容
  3. 标注看不清楚或有歧义的疑问点
  4. 起草初始文档
  5. 前台调用 Codex xhigh 挑刺（详见 Codex调用协议）
  6. 处理反馈后产出最终初始文档，交回主 CLI
```

---

### 2. Opus — 逐页分析

对每张图提取：

```
页面名称：[从文件名或图片推断]
页面路径：[如 /knowledge-base/list]

界面元素：
  顶部区域：[标题、导航、操作按钮]
  主体区域：[列表/表格/卡片/表单]
  底部区域：[分页、确认按钮]
  弹窗/抽屉：[如有，单独描述]

交互逻辑：
  - 点击 [元素] → [发生什么]
  - [条件] 时 [元素] 变为 [状态]

字段定义：
  列表字段：[字段名、是否可排序]
  表单字段：[字段名、类型、必填、字符限制]
  状态枚举：[每个状态值和含义]

疑问点：
  - [图片N · 区域X]：[疑问] → [推断]
```

---

### 3. Opus — 主动补全检查清单

```
空状态 & 异常状态
□ 每个列表是否有空状态设计？
□ 每个操作是否有 loading / 成功 / 失败反馈？
□ 加载失败时显示什么？网络断开时怎么处理？

表单完整性
□ 每个字段的完整校验规则？
□ 提交失败时错误提示的位置和方式？
□ 表单关闭时未保存数据是否需要二次确认？

状态流转
□ 涉及状态的数据，完整状态机是什么？
□ 每个状态转换的触发条件？哪些状态不可逆？

权限差异
□ 不同角色看到的界面是否有差异？
□ 无权限时显示禁用还是隐藏？

原型未画但必须有的功能
□ 批量操作？
□ 分页 / 无限滚动？
□ 搜索 / 筛选的完整参数？
```

### 5. Codex 挑刺 Payload

调用配置见 `references/Codex调用协议.md`（全新会话，xhigh）

```
你是一位资深产品经理，正在 review 一份从原型图分析出来的需求文档草案。
目标是在开发前发现功能遗漏、逻辑矛盾和描述不完整的地方。

【原始需求文档】[原文]
【初始文档草案】[完整内容]

请审查：
1. 功能遗漏 — 原型中体现的功能有无遗漏？原始需求提到但文档未覆盖的？
2. 交互缺口 — 有哪些交互路径文档中没有描述？
3. 状态缺失 — 有哪些状态（加载/空/错误/禁用）未被说明？
4. 字段不完整 — 表单/列表字段是否有缺失的校验规则？
5. 疑问点判断 — 对标注的疑问点，给出你的建议

返回格式：
[页面/章节] - [CRITICAL/HIGH/MEDIUM/LOW] - [问题描述] - [建议]
```

---

### 6. Opus 处理反馈

- CRITICAL/HIGH → 直接修复
- MEDIUM/LOW → 评估后决定，可加入"开放问题"
- 不接受时：resume Codex 会话提交反驳
- 最多 3 轮，超出则上报主 CLI

## 边界处理

| 场景 | 处理方式 |
| ------ | --------- |
| 只有 1 张图 | 正常执行，单页面分析 |
| 图片模糊/不完整 | 标注疑问点，给出推断，继续分析 |
| 原始需求与原型矛盾 | 两者都记录，标注为 CRITICAL 疑问 |
| Codex 不可用 | 跳过挑刺步骤，在确认提示中说明，由人工 review 替代 |
| 原型图超过 20 张 | 按模块分组，分批分析，保持单次不超过 10 张 |

---

## 反模式

- ❌ 只照着图描述，不主动补全逻辑空白
- ❌ 看不清楚的内容直接跳过，不标注疑问
- ❌ 主 CLI 自行分析图片，不委派 Opus
- ❌ 产出后自动进入阶段3
- ❌ 原型与需求矛盾时只取一方，不标注差异
