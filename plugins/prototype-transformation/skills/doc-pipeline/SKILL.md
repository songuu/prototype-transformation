# 前端文档流水线

## 这个 skill 做什么

把一份粗糙的原始需求（文字描述 + 原型图），经过四个阶段的逐步增强，最终产出一份包含界面结构、Figma 设计稿链接、后端接口定义、代码审计报告的**完整前端开发文档**。

每个阶段产出后**人工确认**，确认后才进入下一阶段。

---

## 流程总览

```
原始需求文字 + 原型图
        ↓
阶段2  原型解析   →  初始文档
        ↓  ⛔ 人工确认
阶段3  Figma挂载  →  中间文档
        ↓  ⛔ 人工确认
阶段4  接口集成   →  带接口定义文档
        ↓  ⛔ 人工确认
阶段5  代码审计   →  审计报告
        ↓  ⛔ 人工确认
开发实现
```

---

## 模型分工

| 角色 | 模型 | 负责 |
|------|------|------|
| 主持人（主 CLI） | Sonnet 4.6 | 协调、委派、输出确认提示 |
| 分析者（子代理） | **Opus 4.6** | 阶段2原型分析、阶段4接口绑定 |
| 挑刺者 | gpt-5.3-codex **xhigh** | 阶段2、阶段4文档质量质询 |
| 终审者 | gpt-5.3-codex **high** | 阶段5代码审计 |

> 阶段3（Figma挂载）：主 CLI 直接执行，无需 Opus
> 阶段5（代码审计）：主 CLI 直接调用 Codex，无需 Opus

---

## 阶段入口判断

根据用户提供的内容**自动判断**从哪个阶段进入：

```
提供了原始需求文字 + 图片
    → 进入阶段2

说"已有初始文档，需要挂 Figma" / 提供 Figma 链接
    → 进入阶段3

说"已有中间文档，需要接入接口" / 提供 Swagger/接口文档
    → 进入阶段4

说"审计代码" / "检查需求偏差" / 提供代码路径
    → 进入阶段5

只有文字，没有图片
    → 提示用户上传图片
```

进入对应阶段前，**必须先读取** `references/` 中对应的详细文档：

```
阶段2 → 读取 references/阶段2-原型解析.md
阶段3 → 读取 references/阶段3-Figma挂载.md
阶段4 → 读取 references/阶段4-接口集成.md
阶段5 → 读取 references/阶段5-代码审计.md
需要调用 Codex → 读取 references/Codex调用协议.md
```

## 人工确认机制

每个阶段完成后，主 CLI **必须输出以下提示并完全停止**，不自动推进：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📄 [阶段N · 阶段名] 完成
产物：[文件名]

[本阶段简要统计]

✅ 确认无误 → 回复「确认」进入下一阶段
✏️  有修改意见 → 直接描述，Claude 修订后重新等待确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 产物命名规范

```
阶段2产出：requirements-v1-初始文档.md
阶段3产出：requirements-v2-中间文档.md
阶段4产出：requirements-v3-接口文档.md
阶段5产出：audit-report-[功能名]-[日期].md
修订版本：文件名末尾加 -r2、-r3
```

---

## 升级条件

以下任一情况立即停止并告知用户：

- Codex 迭代达到 3 轮仍无共识
- Codex 不可用（禁止自我证明替代）
- Codex 在不同迭代中给出相互矛盾的指令

---

## 反模式总览

- ❌ 跳过人工确认，自动推进到下一阶段
- ❌ 不读取 references 详细文档就开始执行
- ❌ 主 CLI 自行分析图片，不委派 Opus（阶段2/4）
- ❌ Codex 不可用时自我替代
- ❌ 首次审计传 git diff 而非完整文件
- ❌ 产出后自动修复代码，不等人工确认
