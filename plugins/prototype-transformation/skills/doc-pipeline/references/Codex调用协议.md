# Codex 调用协议

## 各阶段配置速查

| 阶段 | 调用方 | reasoning_effort | 会话策略 | 传入内容 |
|------|--------|-----------------|----------|---------|
| 阶段2 文档挑刺 | Opus 子代理 | `xhigh` | 全新；迭代 resume | 原始需求 + 初始文档草案 |
| 阶段4 接口一致性 | Opus 子代理 | `xhigh` | 全新；迭代 resume | 中间文档 + 接口绑定草案 |
| 阶段5 首次审计 | 主 CLI | `high` | 全新；多批次 resume | 需求文档 + 代码全文（分批）|
| 阶段5 修复复审 | 主 CLI | `high` | resume | git diff |

---

## 调用方式

```
# 全新会话
Skill("codex")
model: gpt-5.3-codex
model_reasoning_effort: [xhigh 或 high]
[Payload]

# 恢复会话
Skill("codex")
Payload 开头写：
  "请恢复上一个 Codex 会话 (resume the previous Codex session)"
[本轮内容]
```

---

## 会话隔离

```
阶段2会话 (Opus持有)   ── 独立
阶段4会话 (Opus持有)   ── 独立
阶段5会话 (主CLI持有)  ── 独立（多批次在同一会话内 resume）
```

---

## 严重程度

| 级别 | 定义 | 处理 |
|------|------|------|
| CRITICAL | 阻塞上线 / 功能方向错误 / 数据损坏 | 必须修复 |
| HIGH | 显著功能偏差 / 明确逻辑错误 | 修复后才能继续 |
| MEDIUM | 存在歧义，不立即导致故障 | 主管判断 |
| LOW | 优化建议 | 记录，不阻塞 |

**MEDIUM/LOW 不阻塞阶段推进。**

---

## 客观 vs 主观

**必须接受：** 功能遗漏、逻辑矛盾、字段不匹配、安全校验缺失  
**可以反驳：** 技术选型偏好、扩展性取舍、"可以更优雅"的建议

---

## 错误处理

- **Codex 不可用** → 停止，告知用户，禁止自我证明
- **resume 失败** → 告知用户，不得用新会话继续
- **前后矛盾** → 停止并列出矛盾，告知用户
- **超过 3 轮** → 停止，告知用户

---

## 回退策略

当 Codex 不可用时，各阶段的降级方案：

| 阶段 | 降级方案 |
| ------ | --------- |
| 阶段2 挑刺 | 跳过 Codex 审查，Opus 自检后产出，确认提示中注明"未经 Codex 审查" |
| 阶段4 一致性审查 | 跳过 Codex 审查，Opus 自检后产出，确认提示中注明 |
| 阶段5 代码审计 | **不可降级** — Codex 是核心审计者，必须停止并告知用户 |

> 降级产出的文档质量低于正常流程，务必在确认提示中明确告知用户。
